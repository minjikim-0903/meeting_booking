# 미팅 예약 시스템 기획서

## 1. 배경 및 문제 정의

- 6~8명(최대 8명 기준 설계, 최소 6명)이 **다음 주 평일 5일** 내로 미팅을 잡아야 함
- 참여자는 4개 직무로 구성: **프로덕트 오너, 개발자, 데이터 분석가, 디자이너**
- 전사 **유연근무제** 적용 중 (출근 08:00-10:00, 퇴근 17:00-19:00, 야근 가능성 존재) → 개인별 근무 시간대가 서로 다름
- 각자 다음을 모두 고려해서 시간을 맞춰야 함
  - 개인 일정 (휴가, 개인 용무 등)
  - 이미 잡혀 있는 다른 회의 일정
  - 갑작스러운 병가 / 당일 연차·반차 발생 가능성
  - 외근
- 미팅을 주도(소집)하는 사람이 고정되어 있지 않음 → **누구나 미팅 소집자가 될 수 있어야 함**
- 결석자를 위한 대안(줌 참여, 회의록 요약 제공)이 시스템에 포함되어야 함

## 2. 목적

**회의실을 예약하는 것**이 최종 산출물이며, 그 예약은 ① 회의실 상황(비어있는지, 수용 인원·시설이 맞는지)과 ② 참여자들의 스케줄(그 시간에 실제로 참석 가능한지)이 동시에 충족될 때만 유효하다. 즉, 이 시스템의 목적은 회의실 상황과 사람들의 스케줄을 함께 고려해서 팀원들의 회의를 잡아주는 것이다.

이를 위해 참여자들의 근무시간·개인일정·회의실 가용 여부를 한 화면에서 조율하고, 초대 → 사전 참석여부 체크 → 자동 확정(회의시간+회의실)까지 이어지는 예약 플로우를 제공한다.

## 3. 회의실 예약플로우 시뮬레이션

현재 구현된 화면을 기준으로 실제 사용자가 밟는 예약 흐름을 순서대로 정리한다. 이모티콘 응답, 본인 일정 직접 등록처럼 구현되지 않은 단계는 제외하고 실제로 동작하는 흐름만 담는다.

```
① 회의실 선택 + 예약 현황 확인 (`/` 단일 페이지)
   별도 회의실 카드 목록 페이지나 `/schedule/[roomId]` 같은 회의실별 라우트는 없음 — 좌측
   "회의실" 컬럼의 드롭다운에서 회의실을 고르면(이름 · 수용 인원 표시) 같은 화면 우측에 이번
   주(오늘 포함) 요일 × 시간대(10분 단위) 그리드가 바로 뜸. 우측 "참석자" 컬럼에서 팀을
   고르면(여러 팀 동시 선택 가능) 해당 팀원 전원이 참석자 목록에 자동 반영되어 참석자 일정이
   캘린더에 겹쳐서 표시됨
        ↓
② 빈 슬롯 클릭 → 예약 요청 다이얼로그
   시작/종료 시간(10분 단위), 회의 목적(필수), 화상회의 링크(선택, 예약자가 URL을 직접 입력),
   참석자(팀 다중 선택 + 개인 선택 병행, 예약자 본인은 기본 포함)를 입력. 선택 인원이 회의실
   수용 인원을 초과하면 경고 표시와 함께 발송 버튼이 비활성화됨
        ↓
③ 예약 요청 발송 → 참석자 전원에게 초대 생성 (상태: 대기중)
   캘린더에는 대기중 슬롯이 대각선 줄무늬로 표시되고, 같은 슬롯에 대한 중복 요청은 막힘
        ↓
④ 초대받은 참여자 응답 (`/requests`)
   사이드바 "받은 요청" 메뉴의 뱃지로 미확인 건수를 먼저 확인 → "요청 온 미팅" 섹션에서
   수락 또는 거절(사유 선택) 버튼으로 응답. 이미 응답한 건은 "지난 미팅" 섹션에서 확인하고
   수락↔거절로 번복도 가능
        ↓
⑤ 응답에 따른 자동 처리
   - 전원 수락 시 → 요청 상태가 자동으로 확정(confirmed)으로 전환, 캘린더에 팀 컬러 단색
     블록으로 표시
   - 한 명이라도 거절 시 → 요청 전체가 거절(rejected) 처리 (부분 인원만 빼고 재조율하는
     자동 로직은 없음 — 소집자가 ⑥의 "일정 변경"으로 다시 요청해야 함)
        ↓
⑥ 예약자 본인의 취소/변경
   본인이 만든 대기중/확정 예약을 캘린더에서 클릭하면 "예약 취소" 또는 "일정 변경"
   ("일정 변경"은 기존 요청을 취소하고, 같은 참석자·목적·시간값을 채운 채로 예약 다이얼로그를
   다시 열어 재요청하는 방식) 가능
        ↓
⑦ 타 팀(본인이 만들지 않은) 예약에 대한 조율 신청
   다른 사람이 만든 예약을 클릭하면 직접 취소·변경은 할 수 없고, 조율이 필요할 때 사유
   메시지(예: "같은 시간대에 급한 임원 보고가 있어 양해 부탁드립니다")를 작성해 "예약 조율 신청"을
   보낼 수 있음 — 단, 실제 알림 발송은 아직 구현되어 있지 않고 화면에 "준비 중" 안내만 표시됨
```

## 4. 사용자 및 역할

| 역할 | 설명 |
|---|---|
| 참여자 (Participant) | 프로덕트 오너 / 개발자 / 데이터 분석가 / 디자이너 / 기획 PM. 초대받은 예약 요청에 수락 또는 거절(사유 선택)로 응답 |
| 소집자 (Organizer) | 미팅 초대를 발송하는 사람. 고정 역할이 아니며 참여자 누구나 될 수 있음 |
| (선택) 타팀 참여자 | 다른 팀 소속 참여자. 본인이 만들지 않은 예약을 클릭하면 조율 신청 메시지를 작성해 보낼 수 있음 (실제 알림 발송은 미구현) |

## 5. 페이지 구성 및 기능 요구사항

### 5.1 회의실 가용 현황 페이지 (최우선 구현)

- 요일(월~금) x 시간대(**10분 단위**) 매트릭스로 회의실별 예약 가능 여부를 시각화
- 회의실별 예약 캘린더 뷰 제공 (별도 페이지 라우팅 없이 `/` 단일 페이지 안에서 회의실 드롭다운으로 전환. 기본으로 이번 주(오늘 포함) 예약 현황을 보여주고, 상단 날짜 이동으로 다음 주 이후도 확인 가능 — 8.10 참고)
- 회의실 정보: 수용 인원, 위치, 모니터(화상회의 장비) 유무를 드롭다운/카드에 표시 (구현: 수용 인원으로 목록 자체를 필터링하지는 않고 전체 회의실을 보여주며, 대신 참석자를 고른 뒤 인원이 수용 인원을 초과하면 경고를 띄우는 방식으로 대체 구현됨 — 5.4 참고)
- 빈 슬롯 클릭 시 바로 예약 플로우(5.4)로 진입 가능

**회의실 예약 불가 기준** (구현 기준 — 아래 중 하나라도 해당하면 그 슬롯에서 새로 예약을 만들 수 없음)

1. **지난 날짜**: "오늘"은 접속 기기의 실제 시각이 아니라, 데모 재현성을 위해 고정한 날짜(`MOCK_TODAY = 2026-07-14`, `mock-data.ts`)를 기준으로 계산됨. 이 날짜보다 이전 날짜의 슬롯은 회색으로 비활성화되어 클릭 자체가 막힘. 다만 날짜 단위 비교라, "오늘" 날짜 안에서 이미 지난 시각(예: 오후에 접속했을 때 오전 슬롯)까지 따로 막지는 않음.
2. **이미 예약이 걸린 슬롯**: 시드 예약(`bookings`)이거나 같은 회의실·날짜·시간대에 겹치는 대기중(pending)/확정(confirmed) 요청이 있으면, 그 슬롯은 새 예약을 만드는 빈 슬롯이 아니라 기존 예약/요청의 정보 다이얼로그로 바로 연결됨 — 같은 슬롯에 중복 예약 자체가 불가능.
3. **지난 일정의 예약/요청**: 위 1번과 마찬가지로 `MOCK_TODAY` 기준으로 날짜가 지난 예약·요청은 캘린더에 회색으로 표시되고, 클릭해도 정보만 보일 뿐 취소/일정 변경/조율 신청 버튼은 숨겨짐(읽기 전용) — 이미 끝난 미팅을 건드릴 수 없게 막는 용도.
4. **(슬롯 클릭은 되지만 발송이 막히는 경우)** 예약 요청 다이얼로그에서 선택 참석자 수가 회의실 수용 인원을 초과하거나(5.4 참고) 회의 목적을 입력하지 않으면 "예약 요청 보내기" 버튼이 비활성화됨.

> **DB 미연결 주의**: 위 2·3번의 "이미 예약됨" 판정은 실제 DB가 아니라 브라우저 `localStorage`(목업 스토어 `mock-booking-store.ts`) 기준이다. **Prisma/Postgres DB는 아직 연결되어 있지 않다** — `prisma/schema.prisma`와 `/api/booking-requests*` 라우트는 존재하지만 현재 화면에서 호출되지 않는다(README "현재는 목업 우선 구현", 6장 참고). 즉 이 중복 예약 방지는 **같은 브라우저 안에서만** 유효하며, 다른 사람이 다른 기기/브라우저로 접속하면 내가 만든 예약이 보이지 않아 같은 슬롯을 중복으로 예약할 수 있다 — 실제 서버 단 동시성 충돌 방지는 되어 있지 않다.

### 5.2 개인 일정 등록 페이지 (미구현)

> 아래는 최초 요구사항이며 화면으로 구현되지 않았다. 현재 참석자 일정은 `mock-participants.ts`의 고정된 목업 데이터로만 존재하고, 사용자가 로그인해서 직접 등록/수정/삭제할 방법은 없다.

- 각 참여자가 본인 계정으로 로그인 후 본인 일정을 직접 등록
- 등록 항목
  - 근무 가능 시간대 (유연근무제 반영: 출근 08:00~10:00 중 선택, 퇴근 시간 자동 계산 또는 별도 입력, 야근 가능 여부)
  - 개인 일정 (휴가/외부 미팅/개인 용무 등 비공개 처리 가능, 시간대만 "불가"로 표시)
  - 기존 확정된 다른 미팅 일정 (연동 또는 수기 등록)
- 상태값 예시: 근무 가능 / 다른 미팅 / 개인 일정(비공개) / 연차 / 반차 / 병가
- 반차/연차/병가는 **당일 갑작스러운 변경**이 가능해야 하므로, 등록 즉시 관련된 예정 미팅에 알림이 가도록 처리 (5.6 참고)

### 5.3 통합 일정 조율 페이지 (참여자 스케줄 통합 뷰)

> 별도 페이지로는 구현되지 않았고, 아래 첫 두 항목만 5.1의 회의실 예약 현황 화면에 통합되어 있다.

- 초대 예정/초대된 참여자 전원의 일정을 한 화면에 겹쳐서(타임라인/그리드) 표시 — **구현**: 참석자를 선택하면 캘린더에 각자의 기존 예약이 겹쳐서 표시됨
- 전원이 동시에 가능한 시간대(교집합)를 자동 하이라이트 — **구현**: 슬롯 단위로 "참석자 전원 가능" 배경색 하이라이트 제공
- 회의실 가용 시간(5.1)과 교집합을 낸 "회의실 O + 전원 가능" 슬롯을 우선 추천 — **미구현**: 별도 추천/정렬 로직 없음 (위 하이라이트가 이미 회의실 캘린더 위에 표시되는 것으로 사실상 겹쳐 보이긴 하지만, "추천"이라 부를 만한 우선순위 로직은 없음)
- 타 팀 참여자가 포함된 경우, 해당 인원의 일정은 별도 소스로 표시하고 조율 난이도를 UI로 구분 — **미구현**: 타 팀 참여자도 동일한 참석자 목록/캘린더에 구분 없이 표시됨

### 5.4 미팅 예약(조율) 페이지

예약 시 확인/입력해야 하는 항목:

1. 회의실 선택 (달력 뷰로 해당 회의실의 가능 날짜 확인) — **구현**
2. 회의 시간 확정 (**10분 단위로 시작/종료 시간 선택**) — **구현**. 단 5.3의 "추천 슬롯" 자동 선택은 없고, 사용자가 빈 슬롯을 직접 클릭해 시작/종료를 고름
3. 참여자 초대 (참여자 일정 자동 조회) — **구현**. 다만 "직무별 최소 6명 ~ 최대 8명" 인원수를 강제로 검증하지는 않고, 선택 인원이 회의실 수용 인원을 초과하면 경고 + 발송 버튼 비활성화로 대체 구현됨
4. 타 팀과의 회의 조율이 필요한 경우
   - 타 팀 참여자 초대 — **구현**: 참석자 피커에서 여러 팀을 동시에 선택해 타 팀 인원을 이 예약 요청에 바로 초대 가능 (8.10 참고)
   - 소집자가 **양해 멘트**를 작성해 함께 발송 — **미구현**: 이 예약 요청 흐름에는 양해 멘트 입력란이 없음. 대신 이미 확정/대기중인 "타인의" 예약을 클릭했을 때 별도의 "조율 신청" 메시지를 보내는 기능만 있음(3장 ⑦, 알림 발송은 미구현)
5. 화상회의 링크 — **부분 구현**: "자동 생성"(Google Meet/Zoom API 연동)은 없고, 예약자가 화상회의 URL을 선택 항목으로 직접 입력
6. 예약 요청 발송 → 참석자 전원에게 초대 생성, `/requests`에서 응답 — **구현** (5.5 참고)

### 5.5 사전 참석 여부 검토 (초대 응답)

- 초대받은 참여자에게 알림 발송 — **부분 구현**: 이메일/슬랙 등 외부 채널 발송은 없음. 대신 사이드바 "받은 요청" 메뉴에 미확인 건수 뱃지가 실시간으로 표시됨(인앱 알림에 해당)
- 참여자 응답 방식 — **구현 다름**: 이모티콘 1탭이 아니라 `/requests` 페이지의 "수락"/"거절" 버튼으로 응답. 거절 시에는 사유를 선택해서 함께 기록. "미정" 상태는 없음 (수락/거절 둘 중 하나이며, 나중에 다시 눌러 번복 가능)
- 응답 현황을 소집자가 실시간으로 확인 — **미구현**: "5/6명 응답, 4명 가능" 같은 응답 현황 요약은 소집자에게 따로 보이지 않음. (예약자가 볼 수 있는 것은 예약 자체의 최종 상태(대기중/확정/거절)뿐)
- **전원 수락 시 자동으로 예약 확정** (5.6) — **구현**
- 응답 마감 시한(예: 24시간) 및 리마인드 알림 — **미구현**: 마감 시한 개념이 없어 응답이 없으면 요청이 대기중 상태로 무기한 남음

### 5.6 예약 확정 및 예외 처리

- 전원 참석 가능 확정 시
  - 회의실 자동 예약 확정 — **구현**: 요청 상태가 자동으로 confirmed로 전환되고 캘린더에 팀 컬러 블록으로 표시
  - 화상회의 링크 확정 및 참여자 전원에게 캘린더 초대 발송 — **부분 구현**: 화상회의 링크가 있으면 확정된 요청 정보에 "화상으로 참석" 버튼이 노출됨. 참여자 전원에게 별도 캘린더 초대(ICS, Google Calendar 등)를 발송하는 기능은 없음
- 일부 불참/충돌 발생 시
  - 자동으로 대안 시간 슬롯 재추천 (5.3 로직 재실행) — **미구현**: 한 명이라도 거절하면 요청 전체가 거절 처리될 뿐, 대안 슬롯을 자동으로 다시 추천하지 않음
  - 소집자에게 재조율 필요 알림 — **미구현**: 별도 알림 없이, 소집자가 직접 예약을 클릭해 상태(거절됨)를 확인하고 "일정 변경"으로 재요청해야 함
- 확정 이후 갑작스러운 변경 발생 시 (병가/당일 연차·반차) — **미구현**: 연차/반차/병가 상태 자체가 시스템에 없어(5.2 참고) 이 흐름 전체가 구현되지 않음
  - 해당 참여자가 상태 변경 시 소집자 및 다른 참여자에게 즉시 알림 — 미구현
  - 결석자를 위한 대체 참여 옵션 제공
    - 화상회의(Zoom/Meet) 링크로 비동기 참여 유도 — **구현**: 확정된 예약에 화상회의 링크가 있으면 누구나 "화상으로 참석" 버튼으로 접속 가능(단, 갑작스러운 결석 감지와 연동된 것은 아니고 링크가 항상 노출되는 것)
    - 회의 종료 후 회의록/요약 자동 공유 대상에 자동 포함 — 미구현 (5.7 참고)

### 5.7 회의록 작성 및 요약 페이지 (미구현)

> 회의록 작성/자동 저장/요약/공유 기능은 전혀 구현되지 않았다. 아래는 최초 요구사항 그대로 남겨둔다.

- 회의 중에는 수기로 회의록 작성 가능 (실시간 메모)
- **회의 종료 시점에 회의록 및 요약본이 자동 저장** (별도 저장 액션 불필요, 회의 상태가 "종료"로 전환되는 순간 트리거)
- 결석자를 고려한 **요약본 자동 생성 및 공유** 기능
- 회의록에 다음 포함
  - 참석자 / 불참자 목록
  - 논의 내용
  - 액션 아이템 (담당자, 기한)
- 회의 종료(자동저장) 즉시 결석자에게 회의록 링크 자동 발송

> - `src/lib/mock-participants.ts` — 참여자(User)와 팀 목록 (하드코딩된 정적 데이터, DB 아님)
> - `src/lib/mock-booking-store.ts` — 예약 요청/초대를 브라우저 `localStorage`에 저장 (`MockBookingRequest`: roomId, 날짜, 시작~종료시간, 목적, 화상회의 링크, 주최자, 상태(pending/confirmed/rejected) / `MockInvitee`: requestId, email, 응답상태(pending/accepted/rejected), 거절 사유)
> - `prisma/schema.prisma`에는 `User`/`Account`/`Session`/`VerificationToken`(Auth.js 표준) + `BookingRequest`/`BookingInvitee`(위 목업과 거의 같은 필드)가 정의돼 있지만, 실제 로그인이 아직 붙지 않아 **현재 화면에서는 전혀 쓰이지 않는다** (README "API 라우트" 참고)
> - `Schedule`(개인 일정), `MeetingRoom`(회의실을 DB에 저장), `Meeting`, `MeetingNote`는 DB든 목업 스토어든 어디에도 없다. 회의실 목록은 `src/lib/mock-data.ts`에 상수로 하드코딩돼 있고, 참여자 개인 일정도 `mock-participants.ts`에 고정값으로만 존재한다(5.2 미구현)

원래 초안이었던 항목:

- **User**: id, 이름, 직무(PO/개발자/데이터분석가/디자이너), 소속 팀
- **Schedule**: user_id, 날짜, 시작~종료시간, 상태(근무가능/다른미팅/개인일정/연차/반차/병가), 비공개 여부
- **MeetingRoom**: id, 이름, 수용인원, 위치, 화상장비 유무
- **RoomBooking**: room_id, 날짜, 시작~종료시간, meeting_id
- **Meeting**: id, 소집자_id, 제목, 상태(초안/검토중/확정/취소), 화상회의 링크, 양해_멘트
- **MeetingInvite**: meeting_id, user_id, 응답상태(가능/불가/미정), 응답시각
- **MeetingNote**: meeting_id, 작성자_id, 내용, 요약본, 액션아이템, 자동저장_시각(회의 종료 시각과 동일)

## 8. 문제 발견 및 개선 방향 (설계 노트)

기획자가 가설을 세우고 설계하면서 스스로 발견한 문제와, 그에 대한 개선 방향을 기록한 노트.

> 아래 8.1~8.4는 최초 작성 버전이며, 8.5에서 개정되었다. 8.5를 최신 기준으로 본다.

### 8.1 가설 (재정리, v1)

팀원의 스케줄과 미팅룸 시간을 확인할 수 있게 하고, 예상치 못한 연차·재택근무 발생 시에는 회의록 요약 공유와 화상미팅으로 참석을 대체할 수 있게 한다.

**1. 팀의 회의를 잡기 위해서 필요한 것**
- 미팅룸 확인
- 각자의 스케줄 확인
- 미팅 가능 시간 확인

**2. 고려해야 할 것**
- 예상치 못한 연차 사용
- 팀원의 타 미팅 시간
- 비어있는 회의룸
- 예약이 되더라도 예상치 못한 미참석 대응
- 회의 목적에 따라 중요도에 따라 스케줄을 조정해야 함

### 8.2 문제 발견 (v1)

1. 미팅을 잡을 목적이라 해도 참석자의 디테일한 스케줄을 공개할 의무가 없다.
2. 미팅을 잡기 위해 팀원이 직접 스케줄을 수정/등록해야 한다.
3. 미팅 시간이 변경되거나 예약이 겹칠 경우 조율이 필요하다.

### 8.3 개선 방향 (v1)

1. **가능/불가능 여부로만 표기한다** — 참석자의 세부 일정 내용은 노출하지 않고, 상태값(가능/불가능)만 초대자에게 보여준다. (문제 1에 대한 대응 → 참석자 일정 확인 기능으로 구현)
2. **HR 시스템과 연동한다** — 연차/반차 등록을 팀원이 이 시스템에 이중으로 입력하지 않도록, 사내 HR 시스템과 연동해 자동 반영한다. (문제 2에 대한 대응, 미구현 — 9장 열린 질문 참고)
3. **조율 버튼 + 우선순위 파악** — 미팅 시간 변경/예약 충돌 시 "조율" 버튼을 통해 회의 목적에 따른 우선순위를 매기고, 중요도가 높은 회의를 우선 확정한다. (문제 3에 대한 대응, 미구현)
4. **회의 목적 우선순위 기준 필요** — 전사적으로 동의된 우선순위 판단 기준이 있어야 조율이 가능함. 화면 우측에 기준을 노출 (기준 항목은 우선 임시로 채워 넣고, 추후 전사 합의 기준으로 교체 예정).

### 8.4 로드맵 반영 (v1)

8.3의 1번(가능/불가능 표기)은 이미 구현 반영됨(참석자 일정 확인 패널). 2~4번은 아직 미구현 상태이며, MVP 로드맵(7장) 이후 단계로 별도 우선순위 조율 기능 추가가 필요함.

### 8.5 가설 · 문제 발견 · 개선 방향 (개정, v2)

**가설**: 팀원의 스케줄과 미팅룸 시간을 확인할 수 있게 하고, 예상치 못한 연차·재택근무 발생 시에는 회의록 요약 공유와 화상미팅으로 참석을 대체할 수 있게 한다.

**1. 팀의 회의를 잡기 위해서 필요한 것**
- 미팅룸 확인
- 각자의 스케줄 확인
- 미팅 가능 시간 확인

**2. 고려해야 할 것** (최종)
- 예상치 못한 연차 사용
- 팀원의 타 미팅 시간
- 비어있는 회의룸
- 예약이 되더라도 예상치 못한 미참석 대응
- 스케줄을 조정 및 조율이 되어야 한다
- 다른 팀/다른 팀원에서도 예약 일정을 확인할 수 있어야 한다
- **시간 제약 없이 어디서든 미팅 스케줄을 확인하고 등록/수정/삭제할 수 있어야 한다** (신규 — 반응형/모바일 접근성 요구. 웹 기반이므로 원칙적으로 충족되나, 별도 반응형 QA는 아직 미실시)

**문제 발견** (최종)

1. 회의실을 잡기 위해 예약 조율, 팀원 스케줄 파악에 드는 리소스(시간·수고)가 많이 든다.
2. 팀원의 스케줄과 미팅룸 예약을 한 페이지에 같이 보여주려 하니 정보 과부하가 생긴다.
3. 미팅을 잡을 목적이라 해도 참석자의 디테일한 스케줄을 공개할 의무가 없다.
4. 미팅을 잡기 위해 팀원이 직접 스케줄을 수정/등록해야 한다.
5. 미팅 시간이 변경되거나 예약이 겹칠 경우 조율이 필요하다.

**개선 방향** (최종)

> 회의실 예약과 개인 스케줄은 서로 다른 기능이므로, 화면에서 각각 별도 컬럼으로 구분한다. (문제 1·2에 대한 대응 → 회의실(좌) / 참석자(우) 2컬럼 레이아웃으로 구현 완료. 회의실 관련 필터는 좌측 회의실 컬럼에, 참석자/팀 필터는 우측에 별도로 배치)

1. **내 팀만 선택해도 회의실 예약 가능 여부를 파악할 수 있게 한다** (팀을 고르면 해당 팀원 전원이 자동으로 참석자 목록에 반영되도록 구현 완료)
2. **HR 시스템과 연동해서 재택, 연차, 외근을 표기한다** (미구현 — 9장 열린 질문 참고)
3. **조율 버튼 + 우선순위 파악** — 미팅 시간 변경/예약 충돌 시 "조율" 버튼을 통해 회의 목적에 따른 우선순위를 매기고, 중요도가 높은 회의를 우선 확정한다 (미구현)
4. **회의 목적 우선순위 기준 필요** — 전사적으로 동의된 우선순위 판단 기준이 있어야 조율이 가능함. 화면 우측에 기준을 노출 (기준 항목은 우선 임시로 채워 넣고, 추후 전사 합의 기준으로 교체 예정, 미구현)

### 8.6 로드맵 반영 (최종)

- 개선 1번은 구현 완료 (팀 선택 시 팀원 자동 반영). "가능/불가능 여부로만 표기"는 참석자 일정 확인 기능으로 이미 구현되어 있어 별도 개선 항목에서는 제외.
- 화면 레이아웃: 좌측 = 회의실(회의실 필터 포함) 컬럼, 우측 = 참석자/팀 선택(별도 필터) 컬럼으로 구현 완료.
- "다른 팀/다른 팀원 예약 확인" 요구는 로그인/세션이 없는 현재 MVP 특성상, 별도 "내 팀" 개념 없이 **예약을 클릭하면 팀·예약자·참석자(팀원)가 항상 보이는 방식**으로 단순화하여 구현 완료.
- HR 연동(재택/연차/외근 표기), 조율 버튼(우선순위), 회의 목적 우선순위 기준은 아직 미구현 — MVP 로드맵(7장) 이후 단계로 별도 진행 필요.

### 8.7 가설 · 문제 발견 · 개선 방향 (개정, v3 — 실사용성 검증 단계)

**배경**: MVP 1~4단계까지는 실제 백엔드(Auth.js Google OAuth 로그인 + Prisma/Postgres 기반 예약 요청 API)를 그대로 붙여 구현했다. 그런데 예약 요청의 전체 사이클(①예약자가 요청 발송 → ②초대받은 사람이 "받은 요청"에서 확인 → ③수락/거절 응답 → ④전원 수락 시 확정)을 **혼자서** 눈으로 끝까지 확인하려면, 예약자 계정과 초대받은 사람 계정으로 각각 로그인해야 했다. 즉 한 사람이 "요청 보내는 화면"과 "요청 받는 화면"을 동시에 검증할 방법이 없어, 기능이 실제로 동작하는지 빠르게 반복 확인하기 어려웠다.

**문제 발견** (이번 세션에서 실제로 화면을 만들며 발견한 것들)

1. 로그인·DB에 의존한 채로는 예약 요청 전체 사이클을 혼자 빠르게 반복 검증할 수 없다 (매번 계정 전환·OAuth 리다이렉트 필요).
2. 예약 요청에 "회의 목적"을 강제로 입력받지 않아, 캘린더에 표시되는 여러 예약이 서로 구분되지 않는다.
3. 예약 가능한 시간이 10분 단위 슬롯 하나로 고정돼 있어, 실제로는 흔한 30분~1시간 단위 회의를 슬롯을 이어붙여 잡을 수 없다.
4. 캘린더에서 "요청은 보냈지만 아직 전원이 응답하지 않은" 예약과 "이미 확정된" 예약이 구분 없이 똑같이 보여서, 다른 사람이 같은 슬롯에 중복으로 예약을 시도할 위험이 있다.
5. "받은 요청"이 몇 건 쌓여 있는지 사이드바에서 바로 보이지 않아, 매번 페이지에 들어가야만 확인할 수 있었다.
6. 지난 날짜의 슬롯도 클릭이 가능해, 실수로 과거 시간에 예약을 시도할 수 있었다.
7. 참석자 목록에서 예약자 본인("나")을 매번 수동으로 체크해야 해서, 정작 예약자 자신이 참석자 목록에서 빠지는 실수가 날 수 있었다.

**개선 방향** (구현 완료)

1. **Mock 세션 + localStorage 기반 목업 스토어 도입** — `src/lib/mock-session.ts`(고정된 가짜 로그인 사용자 `MOCK_USER`), `src/lib/mock-booking-store.ts`(예약 요청/초대 응답을 브라우저 localStorage에 저장하는 "가짜 DB"). 실제 NextAuth/Prisma 백엔드(API 라우트 `src/app/api/booking-requests*`, `prisma/schema.prisma`)는 삭제하지 않고 그대로 둔 채, 프런트엔드(`requests` 페이지, `room-schedule`, `sidebar-nav`, `participant-picker`)만 임시로 이 목업을 쓰도록 전환. 덕분에 한 브라우저 탭에서 예약 요청을 만들고 사이드바 "받은 요청"에서 바로 수락/거절까지 전체 흐름을 혼자 검증할 수 있게 됨. 실제 로그인을 붙이는 시점에 이 두 파일과 관련 import만 걷어내고 기존 API 라우트 호출로 되돌리면 되도록 분리해둠 (문제 1 대응).
2. **회의 목적 입력 필수화** — 예약 요청 다이얼로그에 "회의 목적" 텍스트 입력을 추가하고, 값이 없으면 요청 발송 버튼이 비활성화되도록 막음. 캘린더 슬롯/툴팁에도 목적이 노출되도록 반영 (문제 2 대응).
3. **예약 시간 유연화** — 슬롯 클릭 시 기본 30분 블록을 우선 잡아주되, 같은 회의실·날짜의 앞뒤 기존 예약과 겹치지 않는 범위 안에서 시작/종료 시간을 각각 10분 단위 Select로 자유롭게 재선택할 수 있도록 변경 (문제 3 대응).
4. **캘린더에 대기중/확정 예약 오버레이 표시** — `getActiveRequestsForRoom`으로 해당 회의실의 pending/confirmed 요청을 조회해, 대기중인 슬롯은 대각선 줄무늬 패턴, 확정된 슬롯은 팀 컬러 블록으로 캘린더 위에 직접 표시. 클릭하면 목적·예약자·상태를 툴팁/상세 패널로 확인 가능하며, 같은 슬롯에 대한 중복 요청 클릭을 막음 (문제 4 대응).
5. **사이드바에 받은 요청 미확인 뱃지 추가** — 커스텀 이벤트(`STORE_UPDATED_EVENT`)로 스토어 변경을 구독해, "받은 요청" 메뉴 옆에 대기 중인 초대 건수를 실시간 뱃지로 표시. localStorage의 네이티브 `storage` 이벤트는 같은 탭에서는 발생하지 않아, 별도 커스텀 이벤트로 같은 탭 내 갱신까지 반영함 (문제 5 대응).
6. **지난 날짜 슬롯 비활성화** — 오늘 날짜 이전 슬롯은 클릭 불가 처리(회색 표시 + disabled)하여 과거 시간 예약 실수를 방지 (문제 6 대응).
7. **참석자 피커에서 "나"를 기본 선택 + 항상 최상단 고정** — 참석자 선택 초기값에 `"me"`를 기본 포함하고, 선택된 참석자 목록 정렬 시 예약자 본인이 항상 맨 앞에 오도록 고정. 팀 색상 점과 역할 뱃지도 다른 참석자와 동일하게 노출해 시각적 일관성 확보 (문제 7 대응).

**부수적으로 함께 반영된 것**

- 팀 이름을 특정 회사 느낌이 나는 "그로스팀/리스크팀/결제팀/인증팀"에서 일반화된 "서비스1~5팀"으로 변경. 5번째 팀과 팀원 3명(오하준·배지수·서은우), 새 직무 "기획 PM"을 목업 참여자 데이터(`mock-participants.ts`)에 추가해 팀/인원 커버리지를 넓힘.
- 회의실 카드의 "화상 가능/불가" 뱃지를 제거하고 "모니터 있음/없음" 뱃지만 남겨, 8.5에서 지적한 "정보 과부하(문제 2)"를 한 번 더 줄임.

### 8.8 로드맵 반영 (v3)

- 8.7의 개선 1~7번은 모두 구현 완료. 단, **Mock 세션/스토어 전환은 영구적인 설계 변경이 아니라 실제 로그인 없이도 예약↔응답 전체 흐름을 검증하기 위한 임시 조치**다 (9장 열린 질문에 "걷어내는 시점" 항목 추가).
- HR 연동(재택/연차/외근 표기), 조율 버튼(우선순위), 회의 목적 우선순위 기준, 실제 로그인 전환은 여전히 미구현 — MVP 로드맵(7장) 이후 단계로 별도 진행 필요.
- 8.7로 전체 사이클을 혼자 검증할 수 있게 되자, 검증하는 과정에서 곧바로 이어지는 후속 문제가 드러났다 → 8.9에서 다룸.

### 8.9 가설 · 문제 발견 · 개선 방향 (v3 후속 — 응답 이력 조회 및 자기 예약 관리)

**배경**: 8.7의 Mock 스토어로 예약 요청 → 응답 → 확정 사이클을 혼자 반복 검증할 수 있게 되자, 그 과정에서 "한 번 응답하고 끝"이 아니라 "예약자 입장에서 내가 만든 요청을 취소/변경하고 싶다", "응답자 입장에서 이미 응답한 것도 다시 확인·번복하고 싶다"는 요구가 바로 드러났다. 즉 8.7이 예약 요청의 "생성~1회성 응답"까지만 다뤘다면, 8.9는 그 이후의 **관리(취소/변경/이력 조회)** 단계를 다룬다.

**문제 발견**

8. "받은 요청"에서 한 번 수락/거절하면 화면에서 바로 사라져, 과거에 어떻게 응답했는지 다시 확인하거나 응답을 번복할 방법이 없었다.
9. 캘린더에서 대기중/확정 예약 블록을 클릭해도 정보만 보일 뿐, 예약자 본인이 자신이 만든 요청을 취소하거나 시간을 바꿀 방법이 없었다.
10. (버그성 문제) `requests` 페이지에서 localStorage 목업 데이터를 `useState(() => getInboxForEmail(...))`처럼 컴포넌트 초기화 시점에 바로 읽어오면, 서버 렌더링 결과(항상 빈 값 — localStorage는 서버에 없음)와 클라이언트 첫 렌더링 결과(실제 값)가 달라 React hydration 오류가 날 수 있었다.

**개선 방향** (구현 완료)

8. **"지난 미팅" 섹션 추가** — `getRespondedInvitesForEmail`로 이미 응답한 초대 내역을 별도 섹션으로 표시하고, "수락함"/"거절함" 상태 뱃지와 거절 사유를 함께 노출. "거절로 변경"/"수락으로 변경" 버튼으로 응답을 번복할 수 있게 함 (문제 8 대응).
9. **예약 요청 정보 다이얼로그에 취소/일정 변경 기능 추가** — 캘린더의 대기중/확정 블록을 클릭하면 뜨는 다이얼로그에서, 예약자 본인일 때만 "요청 취소"/"예약 취소"와 "일정 변경" 버튼을 노출. "일정 변경"은 기존 요청을 취소한 뒤 같은 회의실·참석자·목적·시간값으로 예약 요청 다이얼로그를 다시 열어 재요청하도록 구현(`getInviteesForRequest`로 기존 참석자 이메일을 참석자 피커 선택값에 복원). 타인이 만든 요청을 클릭하면 "다른 사람이 예약한 요청이라 직접 취소할 수 없습니다" 안내만 표시 (문제 9 대응).
10. **Hydration 안전한 초기화 패턴으로 변경** — `requests` 페이지의 초기 state를 빈 배열로 두고, 마운트 이후 `useEffect`에서만 `getInboxForEmail`/`getRespondedInvitesForEmail`을 호출해 채우도록 변경 (문제 10 대응, 버그 예방 목적이라 별도 "가설"은 없음).

### 8.10 가설 · 문제 발견 · 개선 방향 (v3 후속 2 — 캘린더 기본 뷰, 타 팀 참여자 초대, 수용 인원 초과)

**배경**: 8.9까지로 예약 요청의 생성~관리 흐름은 다듬었지만, 화면을 실제로 반복 사용하며 캘린더 기본 진입 화면과 참석자 선택 쪽에서 세 가지 문제가 추가로 드러났다.

**문제 발견**

11. 캘린더 기본 진입 화면이 "다음 주" 기준으로 고정돼 있어, 오늘을 포함한 이번 주 예약 현황을 보려면 매번 이전 주로 이동해야 했다. 이 때문에 "지난 날짜는 클릭 불가(회색 처리)" 기능(8.7 문제 6 대응)도 최초 진입 화면에는 회색 처리된 슬롯이 하나도 없어 실제로 동작하는지 눈으로 바로 확인하기 어려웠다.
12. 참석자 피커가 한 번에 한 팀만 선택할 수 있어, 5.4가 요구한 "타 팀과의 회의 조율" 시 타 팀 참여자를 초대하려면 개개인을 일일이 체크해야 했다.
13. 참석자를 회의실 수용 인원보다 많이 선택해도 아무 경고 없이 예약 요청을 보낼 수 있어, 6~8명 규모 회의실에 예를 들어 10명을 초대하는 실수가 가능했다.

**개선 방향** (구현 완료)

11. **캘린더 기본 뷰를 "다음 주"에서 "이번 주(오늘 포함)"로 변경** — `getNextWeekdays` → `getCurrentWeekdays`로 교체. 오늘 이전 날짜가 기본 화면에 바로 보이므로 지난 날짜 비활성화가 최초 진입 시점부터 확인됨. (문제 11 대응. 1장의 "다음 주 평일 5일"은 "회의를 잡아야 하는 임박한 시점"이라는 배경 설명으로 유지하되, 5.1의 캘린더 뷰 설명은 이번 변경을 반영해 "이번 주 기본, 다음 주 이후는 상단 날짜 이동으로 확인"으로 수정)
12. **참석자 피커에서 여러 팀 동시 선택 지원** — 팀 선택 Select 옆 "+" 버튼으로 팀을 추가하면 각 팀의 인원이 모두 참석자 목록에 반영되도록 변경(팀별 Select를 여러 줄로 확장, 각 줄에 제거 버튼 제공). 타 팀 참여자를 개별 체크 없이 팀 단위로 초대할 수 있게 되어 5.4 "타 팀 참여자 초대" 요구를 부분적으로 충족 (양해 멘트 작성은 이 흐름에는 아직 없고, 기존 "예약 조율 신청"(8.x 이전 구현) 흐름에만 존재 — 두 흐름 통합은 미구현). (문제 12 대응)
13. **회의실 수용 인원 초과 경고 추가** — 참석자 피커 하단과 예약 확정 다이얼로그에 선택 인원이 회의실 수용 인원을 초과하면 경고 문구를 표시하고, 초과 상태에서는 예약 요청 발송 버튼을 비활성화. (문제 13 대응)

### 8.11 가설 · 문제 발견 · 개선 방향 (v3 후속 3 — 데모 재현성과 지난 일정 잠금)

**배경**: 8.10까지 캘린더 기본 뷰를 "이번 주"로 바꿨지만, "오늘"을 실제 기기 시각(`new Date()`)으로 계산하고 있어 배포된 데모를 여러 사람이 서로 다른 시점에 열어보면 지난 날짜 회색 처리·기본 뷰가 매번 달라지는 문제가 있었다.

**문제 발견**

14. "오늘"이 실제 기기 시각 기준이라, 데모를 보는 시점에 따라 지난 날짜 회색 처리와 기본 캘린더 뷰가 달라져 재현이 어려웠다.
15. 지난 날짜의 예약/요청 블록을 클릭하면 (날짜만 지났을 뿐) 여전히 취소·일정 변경·조율 신청 버튼이 활성 상태로 남아 있어, 이미 끝난 미팅을 실수로 취소/변경할 수 있었다.

**개선 방향** (구현 완료)

14. **"오늘"을 고정 데모 날짜로 고정** — `MOCK_TODAY`(`mock-data.ts`, `2026-07-14`)를 도입해 실제 기기 시각 대신 이 값을 기준으로 이번 주 계산·지난 날짜 회색 처리를 하도록 변경. 오늘 날짜 컬럼에는 "오늘" 뱃지를 추가해 기준일을 명시적으로 보여줌 (문제 14 대응).
15. **지난 일정을 읽기 전용으로 잠금** — 캘린더의 지난 예약/요청 블록은 팀 컬러 대신 회색으로 렌더링하고, 정보 다이얼로그에서도 취소/일정 변경/조율 신청 버튼을 숨기고 "지난 일정이라 취소/변경/조율 신청을 할 수 없습니다" 안내만 표시 (문제 15 대응).
