# 미팅 예약 시스템 기획서

## 1. 배경 및 문제 정의

- 6~8명 규모의 팀 미팅을 잡아야 하는데, 전사 유연근무제로 참여자마다 근무 시간대가 다르고 각자 개인 일정·기존 회의 일정·외근 등이 있어 다 같이 가능한 시간을 찾기 어렵다.
- 구글 캘린더 같은 기존 캘린더 도구는 이미 시간이 확정된 일정만 예약(등록)할 수 있을 뿐, "몇 명이 언제 다 같이 가능한지 찾는" 조율 과정 자체는 지원하지 않는다.
- 참여자들의 스케줄이 한곳에 모여 있지 않아서, 일정이 새로 필요하거나 변경될 때마다 소집자가 참여자 한 명 한 명에게 직접 연락해 가능 여부를 확인해야 한다. 이 수동 소통 자체가 미팅을 잡는 데 드는 리소스(시간·수고)의 핵심 비용이다.
- 미팅을 주도(소집)하는 사람이 고정되어 있지 않아, 누구나 소집자가 될 수 있어야 한다.

## 2. 목적

**회의실을 예약하는 것**이 최종 산출물이며, 그 예약은 ① 회의실 상황(비어있는지, 수용 인원이 맞는지)과 ② 참여자들의 스케줄(그 시간에 실제로 참석 가능한지)이 동시에 충족될 때만 유효하다. 이를 위해 참여자들의 일정·회의실 가용 여부를 한 화면에서 조율하고, 예약 요청 → 초대 응답 → 자동 확정까지 이어지는 흐름을 제공한다.

## 3. 회의실 예약플로우 시뮬레이션

```
① 회의실 선택 + 예약 현황 확인 (`/` 단일 페이지)
   좌측 "회의실" 컬럼의 드롭다운에서 회의실을 고르면(이름 · 수용 인원 표시) 같은 화면 우측에
   이번 주(오늘 포함) 요일 × 시간대(10분 단위) 그리드가 바로 뜸. 우측 "참석자" 컬럼에서 팀을
   고르면(여러 팀 동시 선택 가능) 해당 팀원 전원이 참석자 목록에 자동 반영되어 참석자 일정이
   캘린더에 겹쳐서 표시됨
        ↓
② 빈 슬롯 클릭 → 예약 요청 다이얼로그
   시작/종료 시간(10분 단위), 회의 목적(필수), 화상회의 링크(선택, 예약자가 URL을 직접 입력),
   참석자(팀 다중 선택 + 개인 선택 병행, 예약자 본인은 기본 포함)를 입력. 선택 인원이 회의실
   수용 인원을 초과하면 경고 표시와 함께 발송 버튼이 비활성화됨
        ↓
③ 예약 요청 발송 → 참석자 전원에게 초대 생성 (상태: 대기중)
   캘린더에는 대기중 슬롯이 대각선 줄무늬로 표시되고, 같은 슬롯에 대한 중복 요청은 막힘
        ↓
④ 초대받은 참여자 응답 (`/requests`)
   사이드바 "받은 요청" 메뉴의 뱃지로 미확인 건수를 먼저 확인 → "요청 온 미팅" 섹션에서
   수락 또는 거절(사유 선택) 버튼으로 응답. 이미 응답한 건은 "지난 미팅" 섹션에서 확인하고
   수락↔거절로 번복도 가능
        ↓
⑤ 응답에 따른 자동 처리
   - 전원 수락 시 → 요청 상태가 자동으로 확정(confirmed)으로 전환, 캘린더에 팀 컬러 단색
     블록으로 표시
   - 한 명이라도 거절 시 → 요청 전체가 거절(rejected) 처리 (소집자가 ⑥의 "일정 변경"으로
     다시 요청해야 함)
        ↓
⑥ 예약자 본인의 취소/변경
   본인이 만든 대기중/확정 예약을 캘린더에서 클릭하면 "예약 취소" 또는 "일정 변경"
   ("일정 변경"은 기존 요청을 취소하고, 같은 참석자·목적·시간값을 채운 채로 예약 다이얼로그를
   다시 열어 재요청하는 방식) 가능. 지난 일정은 읽기 전용(버튼 숨김)
        ↓
⑦ 타 팀(본인이 만들지 않은) 예약에 대한 조율 신청
   다른 사람이 만든 예약을 클릭하면 직접 취소·변경은 할 수 없고, 조율이 필요할 때 사유
   메시지(예: "같은 시간대에 급한 임원 보고가 있어 양해 부탁드립니다")를 작성해 "예약 조율 신청"을
   보낼 수 있음 — 단, 실제 알림 발송은 아직 구현되어 있지 않고 화면에 "준비 중" 안내만 표시됨
```

## 4. 사용자 및 역할

| 역할 | 설명 |
|---|---|
| 참여자 (Participant) | 프로덕트 오너 / 개발자 / 데이터 분석가 / 디자이너 / 기획 PM. 초대받은 예약 요청에 수락 또는 거절(사유 선택)로 응답 |
| 소집자 (Organizer) | 미팅 초대를 발송하는 사람. 고정 역할이 아니며 참여자 누구나 될 수 있음 |
| (선택) 타팀 참여자 | 다른 팀 소속 참여자. 본인이 만들지 않은 예약을 클릭하면 조율 신청 메시지를 작성해 보낼 수 있음 |

## 5. 구현된 기능

### 5.1 회의실 예약 현황

- `/` 단일 페이지에서 좌측 드롭다운으로 회의실을 고르면 우측에 요일 × 시간대(10분 단위) 그리드가 표시됨
- 회의실 정보(수용 인원, 위치, 모니터 유무) 표시
- 기본으로 이번 주(오늘 포함)를 보여주고, 상단 날짜 이동으로 다른 주도 확인 가능. "오늘"은 데모 재현성을 위해 고정된 날짜(`MOCK_TODAY`, `mock-data.ts`)를 기준으로 계산됨

### 5.2 참석자 선택 및 일정 확인

- 팀(서비스1~5팀)을 선택하면 팀원 전원이 참석자 목록에 자동 반영
- "+" 버튼으로 여러 팀을 동시에 선택해 타 팀 참여자도 함께 초대 가능
- 참석자를 고르면 캘린더에 각자의 기존 예약이 겹쳐서 표시되고, 전원이 가능한 슬롯은 하이라이트됨
- 예약자 본인은 기본 선택되며 목록 맨 위에 고정

### 5.3 예약 불가 기준

1. **지난 날짜**: "오늘"은 접속 기기의 실제 시각이 아니라 고정된 날짜(`MOCK_TODAY = 2026-07-14`)를 기준으로 계산됨. 이보다 이전 날짜의 슬롯은 회색으로 비활성화되어 클릭이 막힘.
2. **이미 예약이 걸린 슬롯**: 시드 예약이거나 같은 회의실·날짜·시간대에 겹치는 대기중/확정 요청이 있으면, 새 예약을 만드는 대신 기존 예약/요청의 정보 다이얼로그로 연결됨 — 중복 예약 자체가 불가능.
3. **지난 일정의 예약/요청**: 날짜가 지난 예약·요청은 회색으로 표시되고, 클릭해도 정보만 보일 뿐 취소/일정 변경/조율 신청 버튼은 숨겨짐(읽기 전용).
4. **(슬롯 클릭은 되지만 발송이 막히는 경우)** 선택 참석자 수가 회의실 수용 인원을 초과하거나 회의 목적을 입력하지 않으면 "예약 요청 보내기" 버튼이 비활성화됨.

> **DB 미연결 주의**: 위 2·3번의 "이미 예약됨" 판정은 실제 DB가 아니라 브라우저 `localStorage`(목업 스토어 `mock-booking-store.ts`) 기준이다. **Prisma/Postgres DB는 아직 연결되어 있지 않다.** 즉 이 중복 예약 방지는 **같은 브라우저 안에서만** 유효하며, 다른 사람이 다른 기기/브라우저로 접속하면 내가 만든 예약이 보이지 않아 같은 슬롯을 중복으로 예약할 수 있다 — 실제 서버 단 동시성 충돌 방지는 되어 있지 않다.

### 5.4 예약 요청 생성

- 빈 슬롯 클릭 → 예약 요청 다이얼로그에서 시작/종료 시간(10분 단위), 회의 목적(필수), 화상회의 링크(선택, URL 직접 입력), 참석자를 입력
- 선택 인원이 회의실 수용 인원을 초과하면 경고 표시와 함께 발송 버튼이 비활성화됨
- 발송 시 참석자 전원에게 초대가 생성됨(상태: 대기중). 캘린더에는 대기중 슬롯이 줄무늬로 표시되고, 같은 슬롯에 대한 중복 요청은 막힘

### 5.5 초대 응답 및 자동 확정

- `/requests`에서 "요청 온 미팅"(응답 대기)과 "지난 미팅"(이미 응답함)으로 분리해 표시
- 수락 또는 거절(사유 선택) 버튼으로 응답하며, 이미 응답한 건도 번복 가능
- 전원 수락 시 요청 상태가 자동으로 확정(confirmed)으로 전환되고, 한 명이라도 거절하면 요청 전체가 거절 처리됨
- 사이드바 "받은 요청" 메뉴에 미확인 건수 뱃지가 실시간으로 표시됨
- 화상회의 링크가 있는 확정된 요청에는 "화상으로 참석" 버튼이 노출됨

### 5.6 예약 취소 / 일정 변경

- 본인이 만든 대기중/확정 예약을 클릭하면 "예약 취소" 또는 "일정 변경"(기존 요청을 취소하고 같은 참석자·목적·시간값으로 다이얼로그를 다시 열어 재요청) 가능
- 지난 일정은 읽기 전용으로 취소/변경 버튼이 숨겨짐

### 5.7 타 팀 예약 조율 신청

- 본인이 만들지 않은 예약을 클릭하면 조율 신청 메시지를 작성해 보낼 수 있음(실제 알림 발송은 아직 없고, 화면에 "준비 중" 안내만 표시됨)

## 6. 데이터

- 회의실 목록, 참여자/팀 목록은 `src/lib/mock-data.ts`, `src/lib/mock-participants.ts`에 고정 데이터로 존재
- 예약 요청/초대 응답은 브라우저 `localStorage`(`src/lib/mock-booking-store.ts`)에 저장되는 목업 스토어로 동작
- 실제 DB(Prisma/Postgres)는 연결되어 있지 않다 — 같은 브라우저 안에서만 데이터가 유효하고, 다른 사용자와 실시간으로 공유되지 않는다
